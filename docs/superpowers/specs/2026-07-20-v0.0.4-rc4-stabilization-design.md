## Per-item technical design

### #590 — Cast channel-switch stability

- Add monotonic `_loadGeneration` counter in `flutter_chrome_cast_controller.dart`. Every `loadMedia` bumps it and tags the request; the receiver-status handler drops any status whose generation is stale (< current).
- `iptv_cast_providers.dart:109` — skip `controller.connect(device)` when the target device is already the active connected device; call `load` only.
- Update `iptv_cast_notifier_test.dart:75`: same-device switch expectation becomes `load:next` only (no second `connect:tv-1`).
- New controller test: stale generation-1 status arriving after generation-2 starts must not change session state (issue's AUTO-002).
- TDD: red (update + add tests) → green (implement guard) → refactor.
- Manual verification (UC-001/002, AUTO-003 — Pixel 9 → BRAVIA/Chromecast, `iptv-org` playlist) is a manual step, batched with #889's hardware pass.
- Review: platform-architect (native bridge/cast controller shape) + playback-architect.

### #863 — CI build-web fix

### #7 — Dead-stream UX: honest error + failover

Root cause is two separate gaps, both already have the supporting model layer built and unwired — this is a wiring task, not new infra:

1. **Wrong blame** — `video_player_widget.dart:239` (`streamingState.when(error: (err, _) => _buildError(err.toString()))`) hits the generic `_buildError` path whenever the `streamingStateProvider` stream itself throws, bypassing the existing rich diagnostic pipeline (`platform_media/lib/src/streaming_error_diagnostic_mapping.dart` → `mapStreamingErrorToDiagnostic` → `AiroPlaybackDiagnosticMapper` in `platform_player/lib/src/models/playback_recovery_models.dart`, already rendered correctly by `_buildDiagnosticError` at line 526 when `state.diagnostic` is populated). Fix: ensure the provider always resolves to a `StreamingState` with `diagnostic` set on failure rather than letting the AsyncValue itself go to `error`, so `_buildError` becomes dead code / removed.
2. **No failover on retry** — `VideoPlayerStreamingService.retry()` (`platform_media/lib/src/video_player_streaming_service.dart:508`) just re-calls `playChannel(_state.currentChannel!)` — same channel, same source, forever. `AiroMultiSourceFailoverController` (`platform_player/lib/src/models/multi_source_failover_models.dart:226`) already implements the source-cycling decision logic (`AiroFailoverDecisionCode.switched/ignored/exhausted`) but is never instantiated anywhere in `platform_media` — confirmed zero references. Fix: wire the controller into `playChannel`'s failure path so a fatal, non-retry-eligible diagnostic advances to the next `AiroFailoverSource` before falling back to the current dead-end retry button; surface progress via the already-built `PlayerOverlay._buildFailoverToast` (`feature_iptv/lib/presentation/widgets/player_overlay.dart:254`), which is also unwired today — `_toPlayerViewState` (`video_player_widget.dart:1096`) hardcodes `failover: null` per the known Task 8/9 follow-up in `task-9-report.md`.
- TDD: red (provider test asserting `state.diagnostic` non-null on synthetic 403 + failover controller test for `switched` decision cycling to next source) → green (wire both) → refactor.
- Review: playback-architect (failover controller + diagnostic wiring) + chief-qa-officer (test coverage for both gaps).

### #8 — EPG Settings entry-point gap

Not a missing screen — a missing entry point on one platform only. `app/lib/features/settings/presentation/tv/tv_source_management_section.dart` already composes `XmltvSourceSheet` (line 470) and is reachable via `/settings` → `TvSettingsScreen` on TV, landed pre-rc.3 (`432cdd0c`, polished in `53f3cfac`/#893). The phone `settings_hub_screen.dart` only has a "Playlist Source" tile (`showPlaylistSourceSheet`, line 119) — no XMLTV/EPG source tile — so on Pixel 9 the Guide's own banner ("add an XMLTV source in Settings", `iptv_guide_screen.dart:101`) points at a dead end for phone users specifically.
- Fix: add an "EPG Guide Source" `ListTile` to `settings_hub_screen.dart` alongside "Playlist Source" that opens `XmltvSourceSheet` (already public, already used by the TV variant — no new widget).
- Fix Guide channel ordering: `iptv_guide_screen.dart` currently lists channels in raw ASCII order (`&TV`, `.black` first) — apply the same sort/normalization used elsewhere in channel lists (see `9091`/sentinel group-title normalization work already landed for search).
- Tests: widget test asserting the new tile opens `XmltvSourceSheet` on phone; golden/unit test for Guide channel sort order.
- Review: media-intelligence-architect (EPG) + tv-experience-architect (cross-platform settings parity).

### #9 — Recently Watched pollution

`BoundedRecentListStore.addToRecent` (`platform_history/lib/src/bounded_recent_list_store.dart:29`) is called unconditionally from `iptv_providers.dart:467` and `vod_providers.dart:82` whenever a channel/item is selected — before playback success is known, so a channel that fails to start (403, network error, etc.) still lands in slot 1 of "Jump back in".
- Fix: move the `addToRecent` call to fire only after the streaming service reports a successful play (e.g. on first `StreamingState` transition to `isPlaying == true` for that channel/item), not at selection time.
- Test: fake streaming service that fails to start playback → assert `RecentlyWatchedStorage`/`VodWatchHistoryStorage` unchanged; successful play → assert entry added.
- Review: media-intelligence-architect.
